## 语音播报首次延迟优化（方案1 预热）
- **根因**：`VoiceSpeaker` 每次 `speak` 启动新 QProcess 跑 `piper`，首次时 ONNX 模型（几十MB）+ onnxruntime .so 在 OS 文件缓存冷，从存储读取慢；第二次起在 page cache 快。
- **方案**：`VoiceSpeaker` 加 `Q_INVOKABLE warmup()`——后台异步跑一次 piper（`echo '。' | piper ... > /dev/null 2>&1`，不出声），把模型+.so 加载到 OS page cache。`Main.qml` `Component.onCompleted` 调 `VoiceSpeaker.warmup()`（Qt.callLater 后、登录检查前）。
- **效果**：开机后台预热（~1s），首次真实语音播报时模型已在缓存，省冷缓存读取延迟（~500ms-1s）。合成推理本身（100-500ms）无法省。
- 改 `VoiceSpeaker.h/.cpp` + `Main.qml`，3 文件 lint 通过。
- **piper 模型兼容性**：`zh_CN-xiao_ya-medium.onnx` 与当前 piper 不兼容——其 config 的 phoneme id map 有多字符音素 `"ai"`，piper 要求单 codepoint，报错 `Phonemes must be one codepoint` 崩溃(SIGABRT)。已换回 `zh_CN-huayan-medium.onnx`（兼容可用）。**勿再用 xiao_ya**，除非升级 piper 或找标准 config 版本。

## 重量迟滞算法（Hysteresis）防止临界点跳变
- **背景**：硬件真实值在 X.005 临界点漂移（0.2549 ↔ 0.2551）导致常规四舍五入 0.25/0.26 来回跳，且"显示重量"与"保存/计算重量"用了不同来源的 `WeightManager.netWeight`（3位精度）与 `Math.round(netWeight*100)/100`（2位），数据不一致。
- **方案**：
  1. **`WeightSensor` 加 `Q_PROPERTY displayWeight`**（带迟滞）+ `HYSTERESIS_THRESHOLD = 0.007`（半档 0.005 + 迟滞 0.002）+ `m_displayWeight` 状态 + `displayWeightChanged` 信号。
  2. `consumeBuffer` 在 m_netWeight 更新后用 `netWeight()` getter（已含负20g→0 钳位）做迟滞算法：
     - `round_ = round(raw*100)/100`（常规四舍五入）
     - 上跳：`round_ > m_displayWeight && raw > m_displayWeight + 0.007` → 跳
     - 下跳：`round_ < m_displayWeight && raw < m_displayWeight - 0.007` → 跳
     - 死区 (`D-0.007`, `D+0.007`) 内保持当前显示不变
  3. **`WorkstationPage.qml` 5 处 `WeightManager.netWeight` → `WeightManager.displayWeight`**（所有带重量的计算/显示/保存）：
     - 1004 大字显示 / 31 金额计算 / 1137 保存用 currentWeight / 925 AI 识别 captureVegetable
  4. **不动** 917(`<=0.05` 重量太小不能识别) / 1060(`>8.0` 硬件归零限制) —— 业务阈值判断用原始 netWeight 更精确，不应受迟滞影响。
- **效果**：硬件在 10.003~10.007 死区漂移时显示稳定（不跳），越过死区才跳到下一档；显示/金额/保存三处数据完全一致。
- 改 `WeightSensor.h/.cpp` + `WorkstationPage.qml`，3 文件 lint 通过。
