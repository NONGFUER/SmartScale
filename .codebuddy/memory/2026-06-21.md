## AuthService 登录逻辑重构（去除离线降级）
- **背景**：在线登录成功却被标记为 `mode="OFFLINE"`，导致 PRODUCT_BY_SN 永不触发；且业务失败时降级到离线有安全隐患
- **根因**：`onNetworkReply` 调用 `handleAuthSuccess(..., m_isOnlineMode)`，而 `m_isOnlineMode` 登录前恒为 `false`
- **改动**（`src/services/AuthService.cpp`）：
  1. **admin 特殊处理保留**：`login()` 中 admin 分支仍直接走 `tryOfflineLogin`，仅更新注释
  2. **网络层错误不降级**：`onNetworkReply` 中 `reply->error() != NoError` 分支改为直接 `Q_EMIT loginFailed`，移除 `tryOfflineLogin` 降级
  3. **业务层失败不降级**：`parseAuthResponse` 返回 false（success=false）分支改为直接 `Q_EMIT loginFailed(errMsg)`，移除降级
  4. **online 参数修复**：`const bool online = isRefresh ? m_isOnlineMode : true;` 在线登录成功固定 true，刷新沿用当前模式
- **PRODUCT_BY_SN 触发条件逻辑保持现状**：`handleAuthSuccess` 内 `if (online) tryFetchProductBySn();` 守卫、`tryFetchProductBySn` 内部 `if (!m_isOnlineMode || m_token.isEmpty()) return;` 防御检查均未改动
- **验证**：增量编译通过，AuthService.cpp.o 无 error/warning
- **行为变化**：非 admin 账号在线登录失败（网络/业务）将直接报错给用户，不再静默降级；admin 账号行为不变

## Bug 修复: 图片上传未触发 — recoId 雪花 ID toInt 溢出（又一次）
- **根因**：`onCloudReply` 解析服务器响应时 `remoteId = dataObj.value("recoId").toInt()`，但 `recoId` 是 17 位雪花 ID（如 `"62131312730845184"`，string 类型），远超 int 范围（max 2147483647），toInt 返回 0 → `remoteId > 0` 条件 false → 图片上传被跳过
- **次因**：`custId` 服务器返回空字符串 `""`，原条件 `custId > 0` 不满足，已添加 fallback 到 `userId`
- **最终修复方案（用户决定：不转 64 位，全程保持 QString）**（`src/services/WeightHistoryService.cpp` + `.h`）：
  1. `QString remoteRecoId`，解析用 `dataObj.value("recoId").toVariant().toString()`（兼容 string/number 两种 JSON 格式）
  2. `updateRecordImage` 参数 `recordId` 类型从 `int` 改为 `const QString &`
  3. URL query 直接 `query.addQueryItem("RecoId", recordId)`，不再 `QString::number()`
  4. `custId <= 0` 时 fallback 到 `userId` 字段
  5. cloudId 直接 `model.cloudId = remoteRecoId`
- **教训**：id-type-safety skill 已明确 recoId 是云端雪花 ID，必须 QString，本次仍踩坑。再次提醒：遇到雪花 ID 字段(ingrId/emsId/cateId/recoId/productId) 一律保持 QString，禁止 toInt/toLongLong 转换

## Bug 修复: WeightRecordRepo 多处 prepare 缺 exec() 导致查询恒返回空
- **症状**：`WeightHistoryService::onCloudReply` 打印 "上传成功 id=53 response: {...}" 后再无任何日志，`updateRecordImage`/标记 synced 都不执行，记录永远 `synced=false` 反复重传
- **根因**：`WeightRecordRepo::findById` (108行) `prepare()` + `addBindValue()` 后直接 `q.next()`，**漏了 `q.exec()`**。QSqlQuery::prepare 只是准备 SQL 不执行，next() 没有活动结果集返回 false → 返回默认 WeightRecord(id==0) → `if (model.id > 0)` 整块跳过
- **同 bug 位置**（一并修复）：
  - `findById` (108): SELECT 单条
  - `queryByDateRange` (132): 按日期范围查询
  - `queryByCategory` (150): 按分类查询
  - `queryStatsForDate` (175): 日期统计
- **修复**：每个函数 `addBindValue` 后添加 `if (!q.exec()) { qCritical 打印错误 + 返回空; }`，再进入 `next()` 循环
- **未受影响**：`insert/update/remove` 末尾的 `q.exec()` 原本就正确；`queryAll/queryUnsynced/totalCount/totalWeight` 用 `q.exec("SELECT...")` 直接传 SQL，没问题
- **教训**：Qt QSqlQuery prepare+bindValue 模式必须显式 exec，与直接 exec(sql) 不同
