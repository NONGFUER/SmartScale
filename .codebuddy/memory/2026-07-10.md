## MQTT 设备信息上报修复（app/main.cpp）
- 问题：设备信息(sn/custId)上报只在 `AuthService::userInfoChanged` 触发，且要求 sn 与 custId 同时就绪；但两者异步独立到达（sn 来自 WeightSensor 串口读取的 `snChanged`，custId 来自登录响应）。若登录早于 SN 读回，则 `sn.isEmpty()` 为真直接 return，SN 到达后 `initAndConnect` 也不调 `publishDeviceInfo`，导致设备信息永远空、后端收不到。
- 修复：抽出 `tryPublishDeviceInfo` lambda，同时挂在 `snChanged` 与 `userInfoChanged` 上，两者都齐了才上报，并加诊断日志（打印缺哪个）。
- 注意：admin 离线登录 custId 恒为 0，按设计不上报云端。遗留：initAndConnect 用当时 custId 设遗嘱主题，SN 早于登录时 will 为 cust/0/...，极端情况，未修。

## MQTT 设备信息 payload 字段来源讨论（未改代码，决策阶段）
- 更正：`buildInfoPayload` 实际输出 JSON `{"hardver":"..","softver":"..","sim":".."}`（`MqttClientService.cpp:912`），此前 2026-07-09 记的 `hardver=xxx&softver=xxx&sim=xxx` 拼接格式有误，以 JSON 为准。
- 已确认三字段来源：
  - softver = `SystemInfoService::appVersion()`（格式 `v2.13.2.9_日期`，现成可用）。
  - hardver = 树莓派 5 `/proc/cpuinfo` 的 `Model` 行（"Raspberry Pi 5 Model B Rev 1.1"）。
  - sim = ICCID；但设备 4G 为以太网模式(eth1)、无 ModemManager，现网大概率取不到，需真机验证或走 AT 指令，暂填空串兜底。
- 用户新增需求：payload 除 hardver/softver/sim 外，再加 `revision`(`/proc/cpuinfo` Revision，如 b04171) 和 `serial`(Serial，如 9d075e46a7745a74) 两个字段。
- 计划落点：`SystemInfoService` 构造时读一次 `/proc/cpuinfo`（键名精确匹配 Model/Revision/Serial，`section(':',1).trimmed()`），新增 hardModel/hardRevision/hardSerial 属性；`MqttClientService::buildInfoPayload` 扩为 5 字段；main.cpp 传参。
- 注意区分：payload 的 `serial`=树莓派主板序列号，MQTT topic 用的 SN=称重主板 `WeightSensor::sn()`，两者不同。
- 待确认：新增 JSON 键名（建议全小写 revision/serial）、后端是否支持新字段。

## /proc/cpuinfo 解析硬件信息为空 — atEnd() 虚拟文件坑
- 现象：部署后 `[Main] 硬件信息字段: hardModel="" hardRevision="" hardSerial=""`，MQTT 上报 hardver/revision/serial 全空。
- 根因：`SystemInfoService::parseCpuInfo` 原用 `while (!in.atEnd())` 读 `/proc/cpuinfo`。`/proc/cpuinfo` 是虚拟文件、内核上报 size=0，`QTextStream::atEnd()` 对 size=0 文件可能一开始返回 true，循环体一次不执行 → 三字段全空。
- 修复：改用 `while (!(line = in.readLine()).isNull())` 以 readLine 的 null 返回值判 EOF，不依赖 atEnd()。大小写不敏感匹配 + 去 `\0` 已具备。
- 落点：`src/services/SystemInfoService.cpp`。重新编译部署后查 `data/smartscale.log` 的 `[SystemInfo] cpuinfo key=` 与 `硬件信息解析结果` 确认。
