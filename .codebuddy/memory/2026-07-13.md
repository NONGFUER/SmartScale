## 食材图片缓存失败根因修复

- **现象**：`~/.cache/smartscale/ingr_images/` 目录为空，图片始终未缓存到本地。
- **根因**：接口 `img` 字段是 **S3 预签名 URL**（`https://s3.shxgs.cn:5196/...?X-Amz-Signature=...`，含 `X-Amz-Algorithm`/`X-Amz-Signature` 查询参数，自带查询串认证，且 `X-Amz-Expires=3600` 仅 1 小时有效）。原 `buildImageRequest` 给请求加了 `Authorization: Bearer <token>` 头。AWS S3 规则：**请求一旦带 Authorization 头，便忽略查询串签名、改用头签名验证**，而 Bearer token 不是合法 SigV4 → 返回 403，下载失败。
- **修复**：`buildImageRequest` 中检测 URL 含 `X-Amz-Signature`/`X-Amz-Algorithm` 时**不附加 Bearer 头**；仅非预签名地址才带 token。
- **附带**：URL 扩展名为大写 `.WEBP`，`localImagePathFor` 已用 `toLower()` 匹配，落盘为 `.webp`（原逻辑已覆盖）。
- **注意**：S3 预签名 URL 1 小时过期。`loadFromCache()` 启动时若 `imgLocal` 已落盘则跳过；若首次且 URL 已过期则下载失败，下次 fetch（URL 刷新）会重试。
- **显示注意**：`.WEBP` 能否在 QML `Image` 显示取决于设备 Qt 是否编译 WEBP 插件；缓存落盘不受影响。
- 涉及文件：src/services/UserIngredientService.cpp

## 第二次打开图片仍一张张冒出来（缓存未复用）

- **根因 1**：`CategoryCorrectionDialog.onOpened` 每次打开都 `UserIngredientService.fetchIngredients()`，fetch 回复里 `m_items.clear()` 把内存中的 `imgLocal` 清空，导致重新下载所有图片。
- **根因 2**：`localImagePathFor` 用**完整 URL（含预签名签名）**做 MD5，每次 fetch 生成的预签名 URL 签名不同 → 同一 S3 对象算出不同缓存文件名 → 永远复用不到磁盘已缓存文件。
- **修复**：
  1. `localImagePathFor` 改为对**去掉查询串的 URL** 做哈希（`u.setQuery(QString())`），同一 S3 对象始终映射到同一缓存文件。
  2. fetch 回复清空前，按 `ingrId` 记录已有 `imgLocal`；解析新 item 时若文件仍存在则复用 `imgLocal`，不再重新下载。
- **效果**：重新拉取时内存 `imgLocal` 保留 → QML 直接显示本地文件秒出；`downloadIngredientImages` 检测到文件存在跳过下载。
- 涉及文件：src/services/UserIngredientService.cpp
