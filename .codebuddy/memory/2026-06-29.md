## WiFi 断开后状态显示"已连接" bug 修复
- **根因**：`disconnectWifi()` 成功后，10s 轮询 `refreshWifiStatus()` 读到 nmcli 缓存的 connected 状态将 wifiStatus 刷回 Connected
- **修复1**（之前）：新增 `QElapsedTimer m_disconnectTime` 防回退保护窗口（5秒）
- **根因2（真正根因）**：`refreshWifiStatus()` 中状态解析顺序错误——`"disconnected".contains("connected")` 为 true，导致 `state="30 (disconnected)"` 被错误解析为 `Connected(4)`（子串匹配顺序 bug）
- **修复2**（本次）：将 `disconnected/unavailable` 检查移到 `connected` 前面，确保更长的/更具体的词先匹配

## StatusBar WiFi 图标不显示修复
- **根因**：`iconWifi` 的 visible 条件要求 `cellularStatus !== CellConnected && cellularStatus !== CellRoaming`；当 `refreshCellularStatus()` 走网络接口模式匹配到已连接的 ethernet 设备时，`cellularStatus` 被设为 `CellConnected`(4)，导致 WiFi 图标永远不可见
- **修复**：移除 visible 中对 `cellularStatus` 的限制，简化为 `wifiStatus === Connected` 即显示
- **文件**：StatusBar.qml 第 131-133 行

## WiFi 断开连接后页面渲染崩溃（只剩背景+弹窗）修复
- **现象**：点击断开 WiFi 后，mainLayout（StatusBar+StackView）整棵子树消失，仅剩背景图 + WifiListDialog 弹窗
- **根因**：`onStatusPollTimer()` 通过 `QtConcurrent::run` 在工作线程中调用 `refreshWifiStatus()` + `refreshCellularStatus()`，这些方法内部直接修改 Q_PROPERTY 成员变量（m_wifiSsid/m_wifiIpAddress/m_wifiSignal 等）并 emit 信号到 QML。当用户点击断开（主线程执行 disconnect + 属性清空 + emit）与轮询工作线程几乎同时写入同一批属性时，QML 绑定引擎收到矛盾数据 → 放弃渲染 mainLayout 子树
- **修复方案**：
  1. 新增 `runOnMainThread<Func>()` 模板辅助方法：检测当前线程，非主线程时通过 `QMetaObject::invokeMethod(..., Qt::QueuedConnection)` 投递到主线程事件循环串行执行
  2. 所有 setter 方法（setWifiStatus/setCellularStatus/setLastError/updateHasCellularHardware）改为通过 runOnMainThread 执行
  3. refreshWifiStatus()：将解析结果存到局部变量，最后一次性 runOnMainThread 批量更新所有属性 + 统一发射一次 wifiStatusChanged（避免多次 emit 导致 QML 反复重算）
  4. updateCellularStatusFromNmcli/updateCellularStatusFromMmcli：同上模式
  5. onWifiDisconnectFinished/onWifiConnectFinished：批量属性更新也走 runOnMainThread
- **修改文件**：NetworkManagerService.h（+include functional, +runOnMainThread 声明）、NetworkManagerService.cpp（10处修改）
- **设计原则**：nmcli/mmcli 的 I/O 阻塞操作仍在工作线程执行（不阻塞 UI），只有「成员变量写入 + 信号发射」投递回主线程
