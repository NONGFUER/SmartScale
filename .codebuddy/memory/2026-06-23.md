## 修复 power-monitor 开机日志时间错误（方案 B）

### 背景
`/var/log/power_monitor.log` 里 `[BOOT]` 行的时间戳 = 上次关机时间，而非真实开机时间。
根因：树莓派 VC4_RTC 断电不保持时间，开机时系统时钟被 fake-hwclock 恢复成上次关机时刻，而 `power-monitor.service` 启动过早（仅 `After=local-fs.target`），脚本无脑 `date` 取到错误时间写入日志。后来 NTP 同步修正时钟，但日志已写错。

### 现状诊断
- `/dev/rtc0` 存在但是树莓派片上 VC4_RTC，dmesg 显示 `setting system clock to 1970`（断电归零）
- `systemd-timesyncd` 正常运行，`NTPSynchronized=yes`，当前时间正确
- 脚本 `/usr/local/bin/power_monitor.sh` 极简：`case start/stop` 直接 `date` 写日志

### 实施的方案 B（不改硬件，修服务+脚本逻辑）

**1. 修改 `/usr/local/bin/power_monitor.sh`**
- 新增 `wait_for_ntp_sync()` 函数：轮询 `timedatectl show -p NTPSynchronized --value`，最多等 90 秒
- `start` 分支：先等 NTP 同步，同步成功才写 `[BOOT]`；超时则带 `(NTP not synced, time may be inaccurate)` 标记写入
- `stop` 分支：保持原逻辑（关机时时钟通常是对的）

**2. 修改 `/etc/systemd/system/power-monitor.service`**
- `After=local-fs.target systemd-timesyncd.service`（确保 timesyncd 先启动）
- `Wants=systemd-timesyncd.service`
- 其余保持原样

### 备份
- `/usr/local/bin/power_monitor.sh.bak.20260623`
- `/etc/systemd/system/power-monitor.service.bak.20260623`

### 验证
- 语法检查 `bash -n` 通过
- 用临时 LOG_FILE 测试 start 分支：NTPSynchronized=yes 时立即写入，时间正确
- `systemctl list-dependencies power-monitor.service` 确认依赖链含 `systemd-timesyncd.service`
- 测试时误污染真实日志 2 条（14:12:59 / 14:13:34），已用 `head -n -2` 清理

### 待验证
下次真实重启后，`[BOOT]` 行应写入 NTP 同步后的真实开机时间。
若开机无网络，90 秒超时后写入带标记的错误时间（至少保留记录）。

## StatusBar 布局挤压修复（中间大标题被左右文字挤压）

### 问题
原布局 `[左版权] [fillWidth 占位] [中标题] [fillWidth 占位] [右状态]` 中，三块内容都是自然宽度，左侧版权文字一长就吃掉弹性占位空间，进而挤压中间标题。中间 `Item` 无最小宽度保护，`Text` 用 `anchors.centerIn` 不自适应，被挤变形。

### 修复（`src/ui/components/StatusBar.qml`）
1. 删除左右两个 `Item { Layout.fillWidth: true }` 弹性占位
2. 左版权 `Text` 加 `Layout.maximumWidth: 300` + `elide: Text.ElideRight` 防止无限撑开
3. 中间 `Item` 改 `Layout.fillWidth: true`，内部 `Text` 加 `width: parent.width` + `horizontalAlignment: Text.AlignHCenter` + `elide: Text.ElideRight`，始终居中且不溢出
4. 右侧状态块保持自然宽度

### 布局逻辑变更
```
旧: [左 自然宽度] [fillWidth 占位] [中 自然宽度] [fillWidth 占位] [右 自然宽度]
新: [左 maxW=300+elide] [中 fillWidth 居中] [右 自然宽度]
```
中间标题永远拿走中间所有剩余空间并居中，左右再怎么变长都挤不到它。

### 后续修正：fillWidth 方案标题不真正居中
- **问题**：左右两块宽度不同时，中间 `fillWidth` Item 拿到 `total - leftW - rightW`，其中心 = `(leftW + total - rightW)/2`，只有 `leftW == rightW` 时才等于窗口中心 → 标题偏移
- **修复**：把标题从 RowLayout 抽出来作为 `root` 直接子元素，用 `anchors.centerIn: parent` 锚定到几何中心；RowLayout 内改为 `[左版权][Item fillWidth spacer][右状态]`。标题作为 root 直接子元素渲染在 RowLayout 之上层
- **教训**：RowLayout 的 fillWidth 只能让某块占据"剩余空间"，不能保证其内容相对整个父容器居中——左右宽度不等时必然偏移。要真正居中必须用 `anchors.centerIn: parent` 脱离布局流

## 历史记录水印/选中效果显示最新一条（最终方案）

### 问题
`WorkstationPage.qml` 的 `_selectLatestRecord()` 带 `!root.currentDetailRecord` 守卫，新增称重记录后 `currentDetailRecord` 仍指向旧记录，`historyChanged` 触发但不切换，水印预览和列表选中都停在旧记录上。

### 修改（`src/ui/pages/WorkstationPage.qml`）
1. `_selectLatestRecord()` 保留 `!root.currentDetailRecord` 守卫（仅未选中时选最新，不覆盖用户当前查看项）
2. **关键**：`onPhotoSaved` 中 `addRecord` 调用前先 `root.currentDetailRecord = null`，使 `historyChanged` 触发时守卫通过，自动选中刚写入的最新记录
3. 详情弹窗 `onClosed` 保持 `root.currentDetailRecord = null`（不特地回到最新一条），水印预览靠 `_displayRecord` 回退逻辑显示 `historyEntries[0]`

### 行为效果
- 页面加载 → `currentDetailRecord` 初始 null → 选中最新
- 新增保存记录 → `addRecord` 前清空 → `historyChanged` 触发选中最新（水印+选中同步）
- 手动点列表其他项 → 仍可临时查看该条
- 详情弹窗关闭 → 清空选中，列表无选中项，水印靠回退显示最新

### 设计原则
- `_selectLatestRecord` 保持有条件（守卫 `!currentDetailRecord`），不无条件覆盖
- 详情弹窗关闭不特地回到最新一条，仅清空当前选中
- 新增记录时主动清空以触发选中，而非改 _selectLatestRecord 为无条件

## 归零逻辑重构：归零=硬件去皮（8kg 以上拒绝）

### 背景
原 `WeightSensor::zero()` 用软件偏移（`m_zeroOffset = m_netWeight + m_tareWeight`），与 `tare()` 的硬件去皮（CMD_TARE）是两套独立机制，导致：归零后 UI 不立即变 0（要等下次 consumeBuffer 重算）、软硬件去皮状态可能不一致。

### 用户需求
归零 = 去皮，唯一区别是当前净重 > 8kg 时硬件不支持，需拒绝。

### 修改
**1. `src/hardware/WeightSensor.h`**
- 新增常量 `static constexpr double ZERO_MAX_KG = 8.0;`
- `zero()` 注释更新为「等同硬件去皮，但当前净重 > 8kg 时拒绝」

**2. `src/hardware/WeightSensor.cpp` `zero()`**
- 删除软件偏移逻辑（`m_zeroOffset`/`m_tareWeight` 赋值）
- 改为：`m_netWeight > ZERO_MAX_KG` 时 `Q_EMIT tareDone(false)` 拒绝；否则走 `Q_EMIT requestTare()`（与 `tare()` 完全相同的硬件路径）
- 归零成功后复用 `onTareDone` → QML 显示"去皮成功"

**3. `src/ui/pages/WorkstationPage.qml` 归零按钮**
- 点击时前端先判 `WeightManager.netWeight > 8.0`，超限直接 toast「超过8kg，无法归零」不调 C++
- 8kg 以下调 `WeightManager.zero()` + toast「归零执行中...」

### 注意
- `m_zeroOffset`/`m_tareWeight` 成员保留但归零不再使用，恒为 0.0，`consumeBuffer` 中 `newNetWeight = rawWeightKg` 完全依赖硬件去皮
- 归零成功复用"去皮成功"提示（功能本质相同，可接受）
- 双层拦截：QML 前端即时反馈 + C++ 兜底防止其他调用方绕过
