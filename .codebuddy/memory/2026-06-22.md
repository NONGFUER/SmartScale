# 2026-06-22 工作日志

## 图片上传接口修复 — CustId/RecoId 参数位置错误（form-data body vs URL query）

- **症状**：图片上传接口后端收到 CustId=0, RecoId=0
- **根因**：Swagger 定义 CustId($int64) + RecoId($int64) + File($binary) **均为 multipart/form-data Request Body 字段**，但客户端代码将 CustId/RecoId 放在了 **URL query string** (`?CustId=X&RecoId=Y`) 中。后端从 body 读不到这两个字段 → 默认 0
- **修复**（`src/services/WeightHistoryService.cpp` `updateRecordImage` 约 523 行）：
  1. 移除 `QUrlQuery` / `url.setQuery()` 相关代码
  2. 新增两个 `QHttpPart` 文本字段：`name="CustId"` + `name="RecoId"`，追加到 `multiPart`
  3. File 字段保持不变（已在 form-body 中）
  4. 更新注释说明参数位置
- **附带**：移除了之前临时添加的 10s QTimer 延迟和 `#include <QTimer>`（已定位根因不再需要）
- **教训**：Swagger "No parameters" + Request body 有字段 → 参数在 body 里不在 URL query；对比 Swagger 是最快定位方式

## userId 雪花 ID 溢出修复（又一次踩坑）

- **症状**：称重记录上传 JSON 中 `userId:0`，导致后端拒绝/关联错误
- **根因**：`AuthService` 中 `m_userId` 类型为 `int`（32-bit），解析用 `toInt()`，17 位雪花 ID（如 `61128584684638212`）远超 `int32` 上限 → 溢出为 0
- **次因**：`parseAuthResponse` 中 `outUserId` 形参/out 参数为 `int`，`handleAuthSuccess` 形参也是 `int`，全链路都被截断
- **修复**（`src/services/AuthService.h` + `.cpp` + `WeightHistoryService.cpp`）：
  1. `Q_PROPERTY(int userId ...)` → `Q_PROPERTY(qint64 userId ...)`
  2. `int userId() const` → `qint64 userId() const`
  3. `int m_userId = -1` → `qint64 m_userId = -1`
  4. `handleAuthSuccess(... int userId ...)` → `qint64 userId`
  5. `parseAuthResponse(... int &outUserId ...)` → `qint64 &outUserId`
  6. 解析 `uidVal.toString().toInt()` → `toLongLong()`；`uidVal.toInt(-1)` → `toVariant().toLongLong()`
  7. `WeightHistoryService::buildUploadJson` 中 `json["userId"] = m_authService->userId()` → `QJsonValue(qlonglong(m_authService->userId()))`
  8. `createUserWeightRecord` 同样改 `QJsonValue(qlonglong(...))`
- **skill 更新**：`id-type-safety` skill 中 `userId` 从"业务小整数"列表移到"云端雪花 ID"列表；`SKILL.md` 主文档第 74 行同步更新
- **教训**：再次提醒 — 遇到雪花 ID 字段（ingrId/emsId/cateId/recoId/**userId**/productId）一律 qint64/QString，禁止 toInt

## 全局 Toast 提示组件实现

- **背景**：用户要求针对"保存成功"/"保存失败"等短暂反馈设计可复用非阻塞提示组件，不要为成功失败分别建弹窗
- **新增**：`src/ui/components/Toast.qml`，特性：
  - 4 种状态色（success 绿/error 红/warning 橙/info 蓝）+ 状态图标
  - 队列支持（上限 5 条，超出丢弃最旧），避免短时间大量触发堆积
  - 自动消失（默认 2500ms，error 类 4000ms）
  - 入场：opacity 0→1 + y -height-20→0 平滑过渡（250ms OutCubic）
  - 退场：反向动画，结束 280ms 后取下一条避免视觉重叠
  - MultiEffect 阴影，顶部居中显示 z:9999
- **接入**：
  - `Main.qml` 挂载 `globalToast` 实例并提供 `window.toast(msg, type, duration)` 便捷函数
  - `WorkstationPage.qml` 新增 `Connections` 监听 `WeightHistoryService.cloudSyncSuccess/cloudSyncFailed/userRecordCreated` → 调 `window.toast()` 反馈
- **调用方式**：`window.toast("保存成功", "success")` / `window.toast("保存失败：" + msg, "error", 4000)`
- **验证**：`cmake --build build --target appSmartScale` 通过

## Toast 组件加载失败修复 — 忘记注册到 CMakeLists.txt

- **症状**：新增 `Toast.qml` 后启动白屏，运行日志：`qrc:/qt/qml/SmartScale/src/ui/Main.qml:154:5: Toast is not a type`
- **根因**：项目 `CMakeLists.txt` 用 `qt_add_qml_module` **显式列出** QML_FILES（非自动扫描目录），新建 QML 文件必须手动追加到列表才会被资源系统注册成类型
- **修复**：在 `CMakeLists.txt` 第 46 行后追加 `src/ui/components/Toast.qml`
- **附带优化**：移除 Toast.qml 中 `QtQuick.Effects` / `MultiEffect` 依赖（用 `border` 代替阴影），避免在 AOT 编译环境下的潜在问题
- **教训**：SmartScale 项目新增 QML 文件后**必须**同时更新 `CMakeLists.txt` 的 `qt_add_qml_module` → `QML_FILES` 列表，否则运行时报 "xxx is not a type" 导致整个 Main.qml 加载失败白屏

## Toast 没弹出 — 诊断定位 & 最终修复

- **症状**：保存成功后 `[Toast] 上传成功 id=` console.log 打印了但 Toast 不显示
- **诊断过程**（8 处 `>>> [Diag]` 日志，已清理）：
  - C++ emit ✓ → QML Connections 收到 ✓ → window.toast() 调用 ✓ → _present() 执行 ✓
  - 结论：纯渲染问题，Toast 组件执行了但视觉不可见
- **根因**：Toast.qml 根节点用裸 `Item` + `anchors` 定位到 `ApplicationWindow.contentItem`（QQuickContentItem）。虽然设了 z:9999，但 StackView 页面在同一 contentItem 层级覆盖了 Toast，导致即使 bubble.opacity=1/y=0 也看不到
- **修复**：`Toast.qml` 根节点从 `Item` 改为 Qt Quick Controls 的 `Popup`（原生浮层，天然渲染在窗口 Overlay 最顶层）：
  - 显示控制：`root.open()` / `root.close()` 替代手动 opacity/y 赋值
  - 动画：`enter/exit: Transition`（NumberAnimation opacity + y）替代 Behavior
  - Popup 属性：modal=false, closePolicy=NoAutoClose, padding=0, background=透明 Item
  - 队列逻辑（show/_next/_dismiss/timer）完全保留不变
- **教训**：Qt Quick 中需要全局浮层提示时，必须用 Popup/Dialog 等原生浮层类型，不能用普通 Item+高 z 值——后者在 ApplicationWindow.contentItem 中会被其他子元素（如 StackView）覆盖或裁剪
